# Phase 05.1: Memory Extraction Quality Overhaul - Research

**Researched:** 2026-03-07
**Domain:** n8n memory batch extraction pipeline -- prompt engineering, schema migration, write-time dedup, content format redesign
**Confidence:** HIGH

## Summary

Phase 05.1 overhauls the memory batch extraction pipeline (workflow YOUR_BATCH_EXTRACTION_WORKFLOW_ID) so that Aerys stores human-like memories instead of flat `key: value` database entries. Three tightly-coupled changes are needed: (1) the extraction prompt must filter ephemeral noise and produce memories with conversational context + temporal grounding, (2) new columns on the memories table (`context`, `event_date`, `key_label`) must support the richer format, and (3) a dedup-at-write check must UPDATE existing memories with fresh context instead of creating duplicates.

The retrieval pipeline (YOUR_MEMORY_RETRIEVAL_WORKFLOW_ID) was already fixed pre-phase with dedup-by-key-prefix in the Format Memory Context node and `message_text` passthrough. The retrieval side needs no structural changes -- but the `content` field format it reads will change, so the Format Memory Context node's display formatting should be verified after the extraction changes land.

**Primary recommendation:** Modify the Build Extraction Request node's system prompt, add a Dedup Check node between Parse Observations and Embed Observation, run migration 007 to add `context`, `event_date`, and `key_label` columns, and update Prepare Insert Data + Insert Memory to use the new format. Existing memories (null context/event_date) are preserved -- they decay naturally via the 30% recency weight.

## Architecture Patterns

### Current Pipeline Flow (Memory Batch Extraction)
```
Schedule Trigger (hourly)
  -> Pre-check Message Source (Postgres)
  -> Route to Message Source (IF)
     -> [true]  Fetch Unprocessed Messages (messages table, last 2h)
     -> [false] Fetch from n8n_chat_histories (fallback)
  -> Guard: Skip NULL person_id (IF)
  -> Group Messages (Code, batches of 20)
  -> Build Extraction Request (Code -- LLM prompt)
  -> Call LLM for Extraction (HTTP to OpenRouter)
  -> Parse Observations (Code)
  -> Guard: Skip If Empty (IF)
     -> [not empty] Embed Observation (HTTP)  +  Insert Userinfo (Postgres)
     -> Embed Observation -> Guard: Skip NULL person_id (Insert) (IF)
       -> Prepare Insert Data (Code)
       -> Insert Memory (Postgres -- ON CONFLICT DO NOTHING)
```

### Modified Pipeline Flow (Phase 05.1)
```
Schedule Trigger (hourly)
  -> Pre-check Message Source (Postgres)
  -> Route to Message Source (IF)
     -> [true]  Fetch Unprocessed Messages (messages table, last 2h)
     -> [false] Fetch from n8n_chat_histories (fallback)
  -> Guard: Skip NULL person_id (IF)
  -> Group Messages (Code, batches of 20, NOW GROUPED BY person_id)
  -> Build Extraction Request (Code -- REWRITTEN prompt with filtering + context)
  -> Call LLM for Extraction (HTTP to OpenRouter)
  -> Parse Observations (Code -- UPDATED to extract new fields)
  -> Guard: Skip If Empty (IF)
     -> [not empty]
        -> **Dedup Check (NEW Code node -- queries existing memories by key_label)**
        -> **Route: Insert or Update (NEW IF node)**
           -> [new memory] Embed Observation -> Prepare Insert Data -> Insert Memory
           -> [existing memory] **Update Memory Context (NEW Postgres node)**
        -> Insert Userinfo (Postgres -- unchanged, still parallel from Guard: Skip If Empty)
```

### Key Structural Change: Dedup Check Before Embed
The most significant change is inserting a dedup check BETWEEN Parse Observations and Embed Observation. This:
1. Queries `SELECT id, content, key_label FROM memories WHERE person_id = $1 AND key_label = $2 AND deleted_at IS NULL`
2. If match found: UPDATE that row's `context`, `event_date`, `updated_at` -- skip re-embedding (the core fact hasn't changed)
3. If no match: proceed to Embed + Insert as today

This avoids the OpenRouter embedding API call for duplicates (cost savings + reduced latency).

### Pattern: Group Messages by person_id
Currently Group Messages batches by index (every 20 messages regardless of speaker). This creates mixed-person transcripts that confuse the extraction LLM. The fix groups by person_id FIRST, then batches within each person's messages. This is a one-line change in the Group Messages Code node.

### Anti-Patterns to Avoid
- **Do NOT add semantic dedup (embedding similarity comparison):** The user wants key_label prefix matching, not fuzzy semantic matching. Semantic dedup would be expensive (requires embedding every observation before checking) and could merge distinct memories that happen to be about similar topics. Key_label matching is deterministic and cheap.
- **Do NOT alter the retrieval pipeline's dedup logic:** Retrieval already dedups by key prefix in Format Memory Context. Extraction-time dedup is complementary, not a replacement -- retrieval dedup remains as a safety net.
- **Do NOT wipe existing memories:** Explicit user constraint. Old `key: value` format memories with null context/event_date remain and decay naturally.
- **Do NOT add a separate memory_type column:** The existing `key_label` prefix convention (`basic.`, `user.`, `interest.`, etc.) already encodes memory type. Adding a column is redundant; instead, promote `key_label` to a first-class indexed column on the memories table.

## Schema Design

### Migration 007: Memory Extraction Quality

```sql
-- Add context, event_date, and key_label columns to memories table
ALTER TABLE memories
  ADD COLUMN IF NOT EXISTS context    TEXT,
  ADD COLUMN IF NOT EXISTS event_date TEXT,
  ADD COLUMN IF NOT EXISTS key_label  TEXT;

-- Backfill key_label from existing content (format is "key: value")
UPDATE memories
SET key_label = TRIM(SPLIT_PART(content, ':', 1))
WHERE key_label IS NULL
  AND content LIKE '%:%';

-- Index for dedup lookups
CREATE INDEX IF NOT EXISTS idx_memories_dedup
  ON memories (person_id, key_label)
  WHERE deleted_at IS NULL AND key_label IS NOT NULL;
```

**Column semantics:**
- `context` (TEXT, nullable): Conversational context that produced this memory. Example: "you joked about it when discussing RAM prices". NULL for legacy memories and stable identity facts that need no context.
- `event_date` (TEXT, not TIMESTAMPTZ): Approximate date string from the LLM, e.g. "Feb 28", "last week", "March 2026". Text because temporal precision varies -- sometimes it's a date, sometimes relative. The retrieval pipeline doesn't sort by event_date (it uses created_at + recency score), so this is for display only.
- `key_label` (TEXT, nullable): The category.attribute key, e.g. `basic.location`, `user.vehicle`, `interest.game`. Extracted from the LLM observation and stored as a first-class column for dedup queries. NULL for legacy memories that predate this schema.

**Why TEXT for event_date:** The extraction LLM often produces relative dates ("last week", "yesterday") or approximate dates ("around February"). Trying to parse these into TIMESTAMPTZ would either fail or lose the human-readable quality. The `created_at` column already provides the machine timestamp for recency scoring. `event_date` is for human-readable display in the memory context string.

### Updated Content Format

**Current format (stored in `content` column):**
```
basic.location: Rotonda West, Florida
```

**New format (stored in `content` column):**
```
basic.location: Lives in Rotonda West, Florida
interest.vehicle: White Kia EV6 with Jolteon vinyls -- mentioned while showing off the car (Feb 28)
interest.gaming: Interested in retro game collecting -- came up when discussing childhood favorites
```

The `content` column format becomes: `key_label: fact_text` for stable facts, or `key_label: fact_text -- context (date)` for contextual memories. The `key_label` and `context` and `event_date` columns store the parsed components separately for querying, but `content` remains the single human-readable string that the retrieval pipeline formats for the LLM.

### Display in Retrieval Pipeline

The Format Memory Context node in the retrieval workflow currently formats as:
```
* basic.location: Rotonda West, Florida [discord] (3d ago)
```

After this phase, memories with context will format as:
```
* Lives in Rotonda West, Florida [discord] (3d ago)
* White Kia EV6 with Jolteon vinyls -- mentioned while showing off the car [discord] (5d ago)
```

**Key change:** Strip the `key_label:` prefix from display. The key_label is for dedup, not for the LLM. The retrieval pipeline's Format Memory Context node should be updated to strip everything before the first `:` + space from the content before displaying.

## Extraction Prompt Design

### Current Prompt (in Build Extraction Request)
The current prompt extracts 5 categories (factual_statements, emotional_moments, decisions_made, technical_content, interests) and returns a JSON array with `observation_type`, `key_label`, `value_text`, `privacy_level`, `asserted_by`, `speaker_id_hint`, `confidence`.

### New Prompt Strategy
The prompt needs three changes:
1. **Add filtering rules** to reject ephemeral noise (transient actions, session-specific decisions, meta-commentary)
2. **Add context extraction** -- ask the LLM to include WHY this fact was mentioned and WHEN
3. **Reduce categories** -- remove `emotional_moments` and `decisions_made` (these are the primary noise sources) and replace with a quality bar

### Recommended Prompt (Claude's Discretion)

```
You extract memories from conversation transcripts. Return a JSON array only -- no preamble, no explanation.

## What to EXTRACT (must have lasting value):
- Stable identity facts: name, location, job, relationships, gender, age
- Genuine interests with signal: hobbies, games, media, projects they care about
- Meaningful life events with context: trips, purchases, celebrations, milestones
- Stated preferences for Aerys: how they want to be addressed, communication style
- Vehicle, home, pet details: specific and memorable

## What to REJECT (ephemeral noise -- do NOT extract):
- Transient actions: "going to bed", "grabbing coffee", "brb"
- Session-specific decisions: "parking this bug", "let's do X next"
- Meta-commentary about the system: "the bot is broken", "that workflow failed"
- Timestamps without facts: "it's late", "been a long day"
- Greetings and smalltalk: "hey", "what's up", "how are you"

## Return format:
[{
  "key_label": "category.attribute",
  "value_text": "the fact itself, naturally phrased",
  "context": "why/how this came up in conversation (1 sentence, optional for stable facts)",
  "event_date": "when this happened if mentioned (e.g. 'Feb 28', 'last week', null if not stated)",
  "privacy_level": "public|private",
  "asserted_by": "self|third_party",
  "confidence": 0.0-1.0
}]

## key_label rules:
- Generic prefixes ONLY: basic, user, work, vehicle, interest, relationship, preference, event
- NEVER use a person's name as prefix
- Examples: basic.location, user.vehicle, interest.game, event.trip, preference.communication

## value_text rules:
- Write naturally, as a human would remember it -- NOT as a database entry
- BAD: "Dodge Ram"  GOOD: "Interested in a Dodge Ram truck"
- BAD: "Rotonda West"  GOOD: "Lives in Rotonda West, Florida"
- For interests, include what makes it notable: "Collects retro games, especially SNES titles"

## context rules:
- 1 sentence max, captures the conversational moment
- Include for interests, events, preferences. Skip for basic identity facts.
- BAD: "mentioned in conversation"  GOOD: "came up while joking about RAM prices"
- If the conversation is mundane, context is null

## privacy_level rules:
- "public": name, job, location, hobbies, vehicle, interests, general life facts
- "private": health details, financial specifics, relationship struggles, personal traumas, sexual orientation
- Source channel (DM vs public) is context, not a rule

If nothing worth remembering, return [].
```

### Why This Prompt Works
1. **Explicit REJECT list** prevents the noise that currently pollutes memories. The current prompt extracts `decisions_made` and `emotional_moments` which are mostly session artifacts.
2. **Natural phrasing** in value_text means the memory reads like a human thought, not a database field.
3. **Context field** captures the conversational moment without bloating the main fact.
4. **event_date** is optional and extracted only when actually mentioned -- avoids hallucinated dates.
5. **Reduced observation_type** -- removing the field entirely. The key_label prefix already encodes type, and the LLM doesn't need to classify into a fixed taxonomy to extract good memories.

## Dedup Strategy

### Write-Time Dedup Check

**SQL query for dedup (run in a new Code node):**
```sql
SELECT id, content, context, key_label, created_at
FROM memories
WHERE person_id = $1::uuid
  AND key_label = $2
  AND deleted_at IS NULL
LIMIT 1;
```

**Logic (in Code node):**
```javascript
// For each observation from Parse Observations:
// 1. Query memories for matching person_id + key_label
// 2. If match exists: route to UPDATE path
// 3. If no match: route to INSERT path (embed + insert)
```

**UPDATE path (new Postgres node):**
```sql
UPDATE memories
SET content = $1,
    context = $2,
    event_date = $3,
    updated_at = NOW()
WHERE id = $4::uuid;
```

The UPDATE preserves the original embedding (the core fact hasn't changed semantically -- e.g., "user.vehicle" is still about their vehicle). The context and event_date evolve. If the value_text actually changed significantly (e.g., user got a new vehicle), the new content will be different enough that the next retrieval query may rank the memory differently, but the embedding is still roughly correct for the same key_label domain.

### Why Not Re-Embed on Update?
1. **Cost:** Each embedding call costs tokens via OpenRouter. For a "you mentioned your Kia again" update, the semantic meaning hasn't changed.
2. **Latency:** Avoids an HTTP call per dedup match.
3. **Accuracy:** The key_label domain embedding (`user.vehicle: ...`) is semantically similar regardless of context wording changes.
4. **Exception:** If the value_text itself changes substantially (user traded their Kia for a Tesla), the next extraction will produce a new `value_text` that doesn't match the old content. The key_label matches, but we UPDATE the content AND re-embed. The dedup Code node should compare value_text similarity -- if significantly different, treat as a new memory and re-embed.

### Dedup Threshold: Value Change Detection
Simple heuristic in the Code node:
```javascript
const existingValue = existingMemory.content.split(':').slice(1).join(':').trim();
const newValue = observation.value_text;
// If the core value changed substantially, re-embed
const valueChanged = existingValue.toLowerCase() !== newValue.toLowerCase()
  && !existingValue.toLowerCase().includes(newValue.toLowerCase())
  && !newValue.toLowerCase().includes(existingValue.toLowerCase());
```
- If `valueChanged`: Delete old, insert new (with fresh embedding)
- If NOT `valueChanged`: UPDATE context/event_date only (keep existing embedding)

## Common Pitfalls

### Pitfall 1: n8n Code Node Cannot Make HTTP Calls
**What goes wrong:** Trying to do the dedup SQL query inside a Code node using fetch() or require('http').
**Why it happens:** n8n Code node sandbox blocks network access.
**How to avoid:** The dedup check must be a Postgres node, not inline SQL in a Code node. Use a Postgres node with the aerys credential to query, then route the result through an IF node.
**Warning signs:** "fetch is not defined" or "Cannot find module 'http'" errors in execution log.

### Pitfall 2: Parallel Fan-Out from Guard: Skip If Empty
**What goes wrong:** Currently Guard: Skip If Empty fans out to BOTH Embed Observation and Insert Userinfo in parallel. Adding Dedup Check in between means the fan-out point must change.
**Why it happens:** n8n connection model -- a node's output can feed multiple downstream nodes, but they all run in parallel. Dedup Check must run BEFORE Embed, so the fan-out must be restructured.
**How to avoid:** Remove the direct connection from Guard: Skip If Empty to Embed Observation. Instead: Guard: Skip If Empty -> Dedup Check (Postgres) -> Route: Insert or Update (IF) -> [new] Embed Observation / [existing] Update Memory. Keep Guard: Skip If Empty -> Insert Userinfo as-is (parallel, unaffected by dedup).
**Warning signs:** Embed Observation and Dedup Check both running on every item (wasted embedding calls).

### Pitfall 3: Mixed person_id in Group Messages
**What goes wrong:** Current Group Messages batches by index (every 20 rows). If messages from multiple persons are fetched, a single batch could contain messages from different users, leading to cross-contamination in the extraction LLM (observations attributed to wrong person_id).
**Why it happens:** Both Fetch queries order by created_at, not by person_id.
**How to avoid:** Group by person_id FIRST in the Group Messages Code node. Each output item should contain messages from exactly one person.
**Warning signs:** Memory attributed to wrong person; observation's person_id doesn't match the speaker.

### Pitfall 4: Dedup Postgres Node Returns 0 Rows
**What goes wrong:** When no duplicate exists, the Postgres node returns 0 rows. n8n does not execute downstream nodes when a Postgres node returns 0 rows.
**Why it happens:** This is the same fundamental n8n constraint that required UNION ALL sentinels in the retrieval pipeline (documented in 04-02 summary).
**How to avoid:** Use the UNION ALL sentinel pattern: `SELECT ... UNION ALL SELECT NULL::uuid, NULL, NULL, NULL, NULL WHERE NOT EXISTS(...)`. The downstream Code node filters out the sentinel row.
**Warning signs:** Execution stops at the dedup check for new memories (no duplicate found = 0 rows = chain dies).

### Pitfall 5: key_label Backfill Misparse
**What goes wrong:** The migration backfill `SPLIT_PART(content, ':', 1)` could misparse content that doesn't follow `key: value` format, or content with multiple colons.
**Why it happens:** Legacy memories were always `key_label: value_text` format (verified from current Build Extraction Request prompt), but edge cases exist if the LLM output was malformed.
**How to avoid:** The migration uses `WHERE content LIKE '%:%'` to only backfill rows with a colon. The `SPLIT_PART` extracts everything before the FIRST colon. Verify a few rows manually after migration.
**Warning signs:** key_label values that look like truncated sentences rather than `category.attribute` patterns.

### Pitfall 6: docker exec psql Stalls
**What goes wrong:** Attempting to run migration SQL via `docker exec aerys-postgres-1 psql`.
**Why it happens:** Tachyon kernel 5.4.219 QCM6490 has a cgroup exec hang bug.
**How to avoid:** Create a Manual Trigger + Postgres node workflow in n8n, let the user execute it from the n8n UI. This is the established pattern from all prior phases.
**Warning signs:** Command hangs forever with no output.

### Group Messages by person_id (replacement for current Group Messages node)

// Source: Modified from current batch-group-msgs-01 node
const rows = $input.all().map(i => i.json);

// Group by person_id first
const byPerson = {};

### Dedup Check + Route Logic (new Code node after Postgres dedup query)
```javascript
// This Code node runs after the Dedup Check Postgres query
// It receives the dedup query result and the observation data
const dupResult = $input.first().json;
const obs = $('Parse Observations').item.json;

// Sentinel row check (UNION ALL pattern)
const hasDuplicate = dupResult.id != null;

if (hasDuplicate) {
  // Compare value to decide: update-in-place vs replace
  const existingValue = (dupResult.content || '').split(':').slice(1).join(':').trim().toLowerCase();
  const newValue = (obs.value_text || '').toLowerCase();

  const valueChanged = existingValue !== newValue
    && !existingValue.includes(newValue)
    && !newValue.includes(existingValue);

  return [{
    json: {
      action: valueChanged ? 'replace' : 'update',
      existing_id: dupResult.id,
      person_id: obs.person_id,
      key_label: obs.key_label,
      value_text: obs.value_text,
      context: obs.context || null,
      event_date: obs.event_date || null,
      privacy_level: obs.privacy_level || 'public',
      source_platform: obs.source_platform || 'discord',
      // For 'update': just update context/event_date
      // For 'replace': need to re-embed and insert new + soft-delete old
      new_content: obs.key_label + ': ' + obs.value_text
        + (obs.context ? ' -- ' + obs.context : '')
        + (obs.event_date ? ' (' + obs.event_date + ')' : '')
    }
  }];
} else {
  return [{
    json: {
      action: 'insert',
      person_id: obs.person_id,
      key_label: obs.key_label,
      value_text: obs.value_text,
      context: obs.context || null,
      event_date: obs.event_date || null,
      privacy_level: obs.privacy_level || 'public',
      source_platform: obs.source_platform || 'discord',
      new_content: obs.key_label + ': ' + obs.value_text
        + (obs.context ? ' -- ' + obs.context : '')
        + (obs.event_date ? ' (' + obs.event_date + ')' : '')
    }
  }];
}
```

### Updated Format Memory Context (retrieval pipeline display)
```javascript
// Source: Modified from current mem-ret-format-01 node
// Strip key_label prefix from display for a more natural read
const input = $input.first().json;
const rows = input.rows || [];

if (rows.length === 0) {
  return [{ json: { memory_context: '' } }];
}

const seen = new Set();
const deduped = rows.filter(r => {
  const key = r.content.includes(':')
    ? r.content.split(':')[0].trim().toLowerCase()
    : r.content.trim().toLowerCase();
  if (seen.has(key)) return false;
  seen.add(key);
  return true;
});

const top = deduped.slice(0, 5);

const lines = top.map((r) => {
  // Strip key_label prefix for natural display
  let display = r.content;
  if (display.includes(':')) {
    display = display.split(':').slice(1).join(':').trim();
  }
  const age = r.created_at
    ? `(${Math.round((Date.now() - new Date(r.created_at)) / 86400000)}d ago)`
    : '';
  const src = r.source_platform ? `[${r.source_platform}]` : '';
  return `* ${display} ${src} ${age}`.trim();
});

return [{ json: { memory_context: lines.join('\n') } }];
```

## Implementation Approach

### Recommended Plan Structure: 2 Plans

**Plan 1: Schema Migration + Extraction Prompt + Group Fix**
1. Migration 007: Add `context`, `event_date`, `key_label` columns to memories table with backfill
2. Rewrite Build Extraction Request prompt (the big prompt change)
3. Fix Group Messages to group by person_id
4. Update Parse Observations to handle new JSON format (context, event_date fields)
5. Update Prepare Insert Data to construct new content format
6. Update Insert Memory SQL to include key_label, context, event_date columns
7. Verify: trigger extraction, check new memories have context + proper format

**Plan 2: Write-Time Dedup + Retrieval Display Update**
1. Add Dedup Check Postgres node between Guard: Skip If Empty and Embed Observation
2. Add Route Dedup Result Code node (insert vs update vs replace logic)
3. Add Update Memory Postgres node for the update path
4. Rewire connections: Guard: Skip If Empty -> Dedup Check -> Route -> [insert] Embed | [update/replace] Update Memory
5. Update Format Memory Context in retrieval pipeline to strip key_label prefix
6. End-to-end verification: send test messages, trigger extraction, verify dedup + context + display

### Why 2 Plans, Not 1
The extraction prompt + schema changes can be tested independently from the dedup logic. Plan 1 gives immediate quality improvement (better memories, less noise). Plan 2 adds the dedup optimization. If Plan 1 has issues, they can be caught before adding dedup complexity on top.

## Open Questions

1. **n8n Postgres node: can it do a per-item subquery?**
   - What we know: The dedup check needs to query once per observation. If Parse Observations emits 5 items, we need 5 separate dedup queries.
   - What's unclear: Whether the Postgres node runs once per input item automatically in "executeQuery" mode, or once total.
   - Recommendation: n8n Postgres nodes in executeQuery mode run once per input item when they receive multiple items. This is the standard n8n behavior. However, if batch mode is needed, use SplitInBatches. Test with the first execution.

2. **Dedup check for the n8n_chat_histories fallback path**
   - What we know: The fallback path hardcodes privacy_level='private'. Dedup should work identically on both paths since it checks by person_id + key_label regardless of source.
   - What's unclear: Whether the fallback path even generates meaningful key_labels from chat history format.
   - Recommendation: Apply the same dedup logic to both paths. The Group Messages node already normalizes both paths into the same format before extraction.

3. **max_tokens for extraction LLM**
   - What we know: Currently set to 800. The new prompt asks for more fields per observation (context, event_date).
   - What's unclear: Whether 800 tokens is enough for the richer JSON format.
   - Recommendation: Increase to 1200. The new format adds ~30 tokens per observation (context + event_date). With 5-8 observations per batch, that's 150-240 extra tokens. 1200 provides safe headroom.
