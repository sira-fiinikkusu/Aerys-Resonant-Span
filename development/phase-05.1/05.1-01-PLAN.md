<objective>
Overhaul the memory extraction pipeline's prompt, grouping, and storage format so Aerys stores human-like memories with conversational context instead of flat key-value data.

This plan rewrites the extraction prompt to filter noise and produce richer output, fixes person_id grouping to prevent cross-person attribution, and updates the schema + insert path to store the new format.

Output: Migration 007 SQL file + updated batch extraction workflow with new prompt, grouping, parsing, and insert logic. Existing memories preserved with null context/event_date.
</objective>

**Part A: Create migration 007 SQL file**

Create `~/aerys/migrations/007_memory_extraction_quality.sql` with:
```sql
ALTER TABLE memories
  ADD COLUMN IF NOT EXISTS context    TEXT,
  ADD COLUMN IF NOT EXISTS event_date TEXT,
  ADD COLUMN IF NOT EXISTS key_label  TEXT;

UPDATE memories
SET key_label = TRIM(SPLIT_PART(content, ':', 1))
WHERE key_label IS NULL
  AND content LIKE '%:%';

CREATE INDEX IF NOT EXISTS idx_memories_dedup
  ON memories (person_id, key_label)
  WHERE deleted_at IS NULL AND key_label IS NOT NULL;
```

Then create a Manual Trigger + Postgres node workflow in n8n (using aerys DB credential YOUR_POSTGRES_CREDENTIAL_ID) containing this SQL. Leave the workflow INACTIVE for the user to execute from the n8n UI. **Part B: GET the current batch extraction workflow**

Fetch workflow YOUR_BATCH_EXTRACTION_WORKFLOW_ID via n8n API (or MCP `n8n_get_workflow`). Read the full JSON to understand current node structure, names, and connections before modifying.

**Part C: Rewrite Build Extraction Request node**

Replace the system prompt in the Build Extraction Request Code node with the new prompt from RESEARCH.md (the full prompt under "Recommended Prompt"). Key changes:
- Remove `emotional_moments` and `decisions_made` categories
- Add explicit REJECT list (transient actions, session decisions, meta-commentary, greetings)
- Add `context` and `event_date` fields to the return format
- Remove `observation_type` field (key_label prefix encodes type)
- Remove `speaker_id_hint` field (person_id comes from grouping, not LLM)
- Natural-language `value_text` instructions (not database-style)
- Increase max_tokens from 800 to 1200 to accommodate richer JSON output

**Part D: Fix Group Messages node**

Replace the Group Messages Code node logic to group by person_id FIRST, then batch within each person's messages (max 20 per batch). Use the code from RESEARCH.md "Group Messages by person_id" section. Each output item must contain `messages`, `source_platform`, `source_privacy`, and `person_id` fields.

**Part E: Update Parse Observations node**

Update the Parse Observations Code node to handle the new LLM output format:
- Read `context` and `event_date` from each observation (both nullable)
- Remove `observation_type` and `speaker_id_hint` handling
- Keep `key_label`, `value_text`, `privacy_level`, `asserted_by`, `confidence`
- Build `new_content` string: `key_label + ': ' + value_text + (context ? ' -- ' + context : '') + (event_date ? ' (' + event_date + ')' : '')`
- Pass `key_label`, `context`, `event_date`, and `new_content` downstream

**Part F: Update Prepare Insert Data node**

Update Prepare Insert Data to include key_label, context, and event_date in the merged output alongside embedding. Read the new fields from `$('Parse Observations').item.json` (established pattern from 04-02 -- $json at this node is the Embed output, not the observation).

**Part G: Update Insert Memory SQL**

Update the Insert Memory Postgres node's SQL to include the three new columns:
```sql
INSERT INTO memories (person_id, content, key_label, context, event_date, embedding, source_platform, privacy_level, created_at)
VALUES ($1::uuid, $2, $3, $4, $5, $6::vector, $7, $8, NOW())
ON CONFLICT DO NOTHING;
```
Update queryReplacement to include the new fields in the correct order: `={{ [$json.person_id, $json.content, $json.key_label, $json.context, $json.event_date, $json.embedding_str, $json.source_platform, $json.privacy_level] }}`

**Part H: Deploy updated workflow**

PUT the modified workflow JSON back to n8n via API. Activate the workflow. Export the updated JSON to `~/aerys/workflows/04-02-memory-batch.json`. Commit the export + migration file to the infra repo.

- LLM JSON strip: `text.replace(/^```(?:json)?\n?/m, '').replace(/\n?```$/m, '').trim()` before JSON.parse
- Do NOT use Set nodes (return {} in this n8n version) -- use Code nodes
