<decisions>
## Implementation Decisions

### Memory Format
- Current `key: value` format is a database, not memory — must include context + timestamps
- New format: the fact + conversational context it came from + when it happened
- Example: "Dodge Ram — you joked about it when discussing RAM prices (Feb 28)"
- Stable identity facts (location, vehicle, gender) can remain simple — context is optional for these

### Extraction Prompt Filtering
- EXTRACT: stable identity facts, genuine interests with signal, meaningful life events with context, stated preferences for Aerys
- REJECT: transient actions (going to bed), session-specific decisions (parking a bug), things needing timestamp/context to be meaningful, meta-commentary about the system
- Events CAN be memories IF they have context: "you told me last week about going to MRT, how are the mimosas?"

### Dedup at Extraction Time
- Before inserting a new memory, check if a memory with the same key prefix exists
- If duplicate found: UPDATE with fresh context rather than creating duplicate #68
- Turns stale entries into living memories that gain context over time

### Existing Memories — DO NOT NUKE
- User explicitly does not want a full wipe
- Old memories stay with null context/event_date fields
- They'll naturally decay via the 30% recency weight in retrieval scoring as better-formatted memories accumulate

### Implementation Discretion
- Schema changes to memories table (context column, event_date column) — implementation approach
- Exact extraction prompt wording
- How to structure the dedup check (SQL query pattern, threshold for "same memory")
- Whether to add a memory_type column or use the existing key prefix convention

</decisions>
