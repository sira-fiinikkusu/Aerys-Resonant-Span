## Decisions Made
- CTE soft-delete merged into Insert Memory — eliminates race condition between parallel soft-delete and insert on unique `(person_id, key_label) WHERE deleted_at IS NULL` index
- Pre-Embed Context added as identity pass-through before Embed Observation — HTTP Request nodes replace all item JSON with API response, breaking upstream references
- Guard: Skip NULL person_id uses `.first()` not `.item` — avoids pairedItem tracking failure when Code node `.map()` creates new items across IF node branching
- Update Memory confirmed working despite showing 1 output item — Postgres executeQuery collapses UPDATE results but executes all queries; safe for terminal nodes

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Route Dedup Result only processed first item**
- **Issue:** Used `$input.first()` — processed only 1 of N observations
- **Fix:** Changed to `$input.all()` with `.map()` and index-based `$('Parse Observations').all()` matching with `pairedItem: {item: idx}`

**2. [Rule 1 - Bug] Guard: Skip NULL person_id pairedItem ambiguity**
- **Issue:** `$('Parse Observations').item` threw "Multiple matching items" after `.map()` broke pairedItem chain
- **Fix:** Changed to `$('Parse Observations').first().json.person_id` — safe because all items share same person_id (grouped upstream)

**3. [Rule 1 - Bug] Soft Delete null UUID cast**
- **Issue:** Insert items had `existing_id: null` causing UUID cast error
- **Fix:** Added dummy UUID fallback `$json.existing_id || '00000000-0000-0000-0000-000000000000'`

**4. [Rule 2 - Restructure] Soft Delete Postgres node dropped items with 0-row results**
- **Issue:** Postgres executeQuery silently drops items when UPDATE affects 0 rows — insert items had no matching row to delete, so Soft Delete dropped them
- **Fix (iteration 1):** Added Route: Replace or Direct Insert IF node to bypass Soft Delete for inserts
- **Fix (iteration 2):** Removed separate Soft Delete node entirely; merged into Insert Memory CTE

**5. [Rule 2 - Restructure] Pre-Embed Context received Postgres output instead of observation data**
- **Issue:** Soft Delete (Postgres) replaced all JSON with `{success: true}`, so Pre-Embed Context received wrong data
- **Fix:** Added Pre-Embed Context Code node at the merge point before Embed — identity pass-through preserving all observation fields

**6. [Rule 2 - Restructure] Insert Memory race condition on unique index**
- **Issue:** Parallel Soft Delete and Insert Memory raced on unique `(person_id, key_label)` constraint — insert tried before delete completed
- **Fix:** Merged Soft Delete into Insert Memory as CTE: `WITH soft_del AS (UPDATE ... RETURNING id) INSERT INTO ...` — atomic single-query operation in same transaction

**7. [Rule 3 - Non-bug] Update Memory appeared to process only 1 of N items**
- **Issue:** Postgres executeQuery output showed 1 item for 3 inputs — appeared broken
- **Resolution:** Verified via database query — all 3 UPDATEs executed correctly. Postgres collapses UPDATE results to single output with pairedItem referencing all inputs. Not a bug since Update Memory is terminal.

## Issues Encountered
- n8n Postgres executeQuery nodes silently drop items when query affects 0 rows (undocumented behavior)
- n8n HTTP Request nodes (Embed Observation) replace all item JSON with API response — must stash data upstream
- pairedItem tracking breaks when Code nodes use `.map()` to create new items across IF node branching paths
- Parallel Postgres nodes race on unique constraints — must use CTE for atomicity
