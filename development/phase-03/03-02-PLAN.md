**Telegram adapter (02-02) — resolver + !command detection + user commands:**

Telegram keeps text-prefix commands. Modify 02-02 to insert command detection, resolver, and command handlers between "Normalize Message" and the Core Agent call:

**Node A: Detect Command** (Code node)

```javascript
const msg = $input.item.json;
const text = (msg.message_text || '').trim();

const isCommand = text.startsWith('!');
let command = null;
let commandArgs = [];

if (isCommand) {
  const parts = text.slice(1).trim().split(/\s+/);
  command = parts[0].toLowerCase();    // 'link', 'unlink', 'profile', 'status'
  commandArgs = parts.slice(1);
}

return [{
  json: {
    ...msg,
    is_command: isCommand,
    command: command,
    command_args: commandArgs
  }
}];
```

**Node B: Route** (Switch node)
- Output 0 (Commands): `{{ $json.is_command === true }}`
- Output 1 (Chat): `{{ $json.is_command !== true }}`

**Chat path** (Output 1):
- **Resolve Identity** (Execute Sub-workflow, 03-01, waitForSubWorkflow: true)
  - Input: `{platform: $json.platform, platform_user_id: $json.user_id, username: $json.username}`
- **Merge person_id** (Set node) — merge resolver output into payload
- → Existing Core Agent Execute Sub-workflow call

**Commands path** (Output 0):
- **Resolve Identity** (same sub-workflow — commands also need a person_id)
- **Merge person_id** (Set node)
- **Command Router** (Switch node on `$json.command`): `link`, `unlink`, `profile`, `status`

**!status handler:**
```sql
-- Postgres Execute Query: Get linked accounts
SELECT pi.platform, pi.platform_user_id, pi.username, p.display_name
FROM platform_identities pi
JOIN persons p ON p.id = pi.person_id
WHERE pi.person_id = $1;
```
Query Parameters: `={{ [$json.person_id] }}`

Format reply (Code node): "Your identity: **[display_name]**. Linked platforms: discord, telegram."
Reply via Telegram node (Send Message, chat_id: `$json.chat_id`).

**!profile name \<name\> handler:**

- `UPDATE persons SET display_name = $1 WHERE id = $2`
- Query Parameters: `={{ [command_args.join(' '), person_id] }}`
- Reply: "Display name updated to: [name]"

**!link handler (issue OR redeem):**

Branch on `command_args.length`:
- 0 args → **Issue flow**
- 1 arg → **Redeem flow**

Issue flow:

Reply: "Your link code is: **[CODE]**\n\nRun `/link code:[CODE]` in Discord within 10 minutes."

Redeem flow:
```sql
-- Lookup code
SELECT pl.person_id::text as source_person_id, pl.platform as source_platform
FROM pending_links pl
WHERE pl.code = $1 AND pl.expires_at > NOW();
```
Query Parameters: `={{ [$json.command_args[0].toUpperCase()] }}`

- If not found → Reply: "Code not found or expired. Run `!link` again to get a new code."
- If same person_id → Reply: "These accounts are already linked."
- If different persons → Execute merge (6 sequential Postgres Execute Query nodes):
  - A: `UPDATE platform_identities SET person_id = $1 WHERE person_id = $2` (winner=source, loser=current)
  - B: `UPDATE conversations SET person_id = $1 WHERE person_id = $2`
  - C: `UPDATE messages SET person_id = $1 WHERE person_id = $2`
  - D: `UPDATE memories SET person_id = $1 WHERE person_id = $2`
  - E: `UPDATE persons SET deleted_at = NOW() WHERE id = $1` (loser)
  - F: `DELETE FROM pending_links WHERE code = $1`
  - Each step uses array Query Parameters (avoid comma-split bug — RESEARCH.md Pitfall 1)
- Reply success: "Accounts linked! Your Telegram and Discord identities are now unified."

Discord slash command interactions are POSTed by Discord to `https://your-domain.example.com/webhook/discord/aerys-commands`. This task creates the webhook workflow that handles them, registers the commands with Discord, and completes the Discord Interactions Endpoint URL configuration.

**Part A: Create "Aerys Discord Slash Commands" webhook workflow**

Create a new n8n workflow via API with the following node chain:

1. **Discord Webhook** (n8n-nodes-base.webhook)
   - Method: POST
   - Path: `discord/aerys-commands`
   - Response Mode: `responseNode` (holds connection open for RespondToWebhook)
   - Options: `rawBody: true`

2. **ParseInteraction** (Code node)
```javascript
const body = $input.item.json.body || $input.item.json;

if (body.type === 1) {
  return [{ json: { isPing: true } }];
}

if (body.type === 2) {
  const data = body.data || {};
  const user = body.member?.user || body.user || {};
  const options = data.options || [];
  const guild_id = body.guild_id;

  const optionValues = {};
  for (const opt of options) {
    optionValues[opt.name] = opt.value;
  }

  return [{
    json: {
      isPing: false,
      commandName: data.name || 'unknown',
      userId: user.id,
      username: user.username || user.global_name || 'Unknown',
      guild_id: guild_id,
      interaction_id: body.id,
      interaction_token: body.token,
      options: optionValues
    }
  }];
}

return [{ json: { isPing: false, error: 'Unknown interaction type' } }];
```

3. **IsPing** (IF node)
   - Condition: `$json.isPing === true`
   - True → **RespondPing**
   - False → **RouteCommand**

4. **RespondPing** (Respond to Webhook)
   - Body: `{"type":1}`
   - Content-Type: application/json
   - Status: 200

5. **RouteCommand** (Switch node on `$json.commandName`)
   - `link` → Link handler chain
   - `unlink` → Unlink handler chain
   - `profile` → Profile handler chain
   - `status` → Status handler chain
   - default → **RespondUnknown**

**For each command, the first node is always Resolve Identity (Execute Sub-workflow, 03-01 Identity Resolver, waitForSubWorkflow: true):**
- Input: `{platform: 'discord', platform_user_id: $json.userId, username: $json.username}`
- Follow with **Merge person_id** (Set node) to carry person_id + original interaction data forward

**/status handler:**
```sql
SELECT pi.platform, pi.platform_user_id, pi.username, p.display_name
FROM platform_identities pi
JOIN persons p ON p.id = pi.person_id
WHERE pi.person_id = $1;
```
Query Parameters: `={{ [$json.person_id] }}`

Format (Code node) → **RespondStatus** (Respond to Webhook):
```javascript
const rows = $input.all().map(i => i.json).filter(r => r.platform);
const displayName = rows[0]?.display_name || 'Unknown';
const linked = rows.map(r => `${r.platform} (${r.username || r.platform_user_id})`).join(', ');
return [{
  json: {
    discordResponse: {
      type: 4,
      data: {
        content: `**Identity:** ${displayName}\n**Linked platforms:** ${linked || 'none'}`,
        flags: 64
      }
    }
  }
}];
```
RespondToWebhook body: `={{ JSON.stringify($json.discordResponse) }}`

**/profile handler:**
- options.name is the new display name (required option in registration)
- `UPDATE persons SET display_name = $1 WHERE id = $2`
- Query Parameters: `={{ [$json.options.name, $json.person_id] }}`
- **RespondProfile** (Respond to Webhook): `{"type":4,"data":{"content":"Display name updated to: **[name]**","flags":64}}`

**/link handler:**

Branch (IF node): `$json.options.code` is empty or undefined → Issue flow / Redeem flow

**Issue flow:**
```javascript
// Code node: Generate Code (Math.random — crypto not available per n8n sandbox)

for (let i = 0; i < 6; i++) {

}

return [{ json: { ...($input.item.json), verification_code: code, expires_at: expiresAt } }];
```
Store code (Postgres):
```sql
INSERT INTO pending_links (code, person_id, platform, expires_at)
VALUES ($1, $2, 'discord', $3);
```
Query Parameters: `={{ [$json.verification_code, $json.person_id, $json.expires_at] }}`

**RespondLinkIssue** (Respond to Webhook):
```json
{"type":4,"data":{"content":"Your link code is: **[CODE]**\n\nRun `!link [CODE]` in Telegram within 10 minutes.","flags":64}}
```

**Redeem flow:**
```sql
SELECT pl.person_id::text as source_person_id, pl.platform as source_platform
FROM pending_links pl
WHERE pl.code = $1 AND pl.expires_at > NOW();
```
Query Parameters: `={{ [$json.options.code.toUpperCase()] }}`

- Not found → RespondToWebhook: `{"type":4,"data":{"content":"Code not found or expired. Run \`!link\` in Telegram to get a new code.","flags":64}}`
